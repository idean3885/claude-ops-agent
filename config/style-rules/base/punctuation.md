# 한국어 구두점 (base)

한국어 글에서 구두점은 "끊어 읽어야 의미가 살 때"만 쓴다.
시각적 호흡 장식은 구두점의 역할이 아니다.
`readability.md K2`(합쇼체 통일) 및 `tone.md T1`(em dash 금지)과 함께 문장 리듬의 규범을 이룬다.

---

## PN1. 짧은 부사절 뒤 쉼표 생략

시간·조건·이유를 이끄는 짧은 부사절 뒤에는 쉼표를 찍지 않는다.
렌더링된 한 줄 이내이고 주절과의 중의성이 없으면 생략이 기본값이다.

> **근거**: 국립국어원 「한글 맞춤법」 부록 "문장 부호"는 부사절 뒤 쉼표를 의무로 두지 않는다.
> 호흡 보조 목적의 관습적 쉼표는 한국어 산문의 간결성을 해친다.

Bad:
```markdown
Forward Auth는 API Gateway가 요청을 백엔드로 전달하기 전에, 외부 인증 서비스에 판정을 위임하는 방식입니다.
파일 요청이 들어오면, Gateway는 Auth에 서브리퀘스트를 보냅니다.
```

Good:
```markdown
Forward Auth는 API Gateway가 요청을 백엔드로 전달하기 전에 외부 인증 서비스에 판정을 위임하는 방식입니다.
파일 요청이 들어오면 Gateway는 Auth에 서브리퀘스트를 보냅니다.
```

**예외**: 부사절이 길거나 주절의 주어와 중의성이 발생할 때는 쉼표를 유지한다.

## PN2. 연결어미 `-고` 뒤 쉼표 제한

연결어미 `-고`(`~하고`, `~지 않고`, `~되고`)로 이어지는 동사구 대등 접속에서는 쉼표를 기본적으로 생략한다.
`-고`는 이미 접속의 역할을 수행하므로, 뒤에 쉼표를 덧붙이면 호흡 장식으로만 기능한다.

**쉼표 유지가 정당한 경우**: `~이 아니라, ~이다` / `~이 아닌 게 아니라, ~` 같이 **명시적 대조 어휘**가 있어 대조의 의미 경계를 표시할 필요가 있을 때.

> **연계**: `ai-tells.md C-5` (연결어미 뒤 쉼표)는 이 규칙의 S1 카운터파트.

Bad (장식성 쉼표):
```markdown
Gateway가 직접 인증·인가를 처리하지 않고, 별도 서비스에 위임하는 구조입니다.
요청을 수신하고, 판정을 받은 뒤 백엔드로 전달합니다.
```

Good (쉼표 없이 자연스럽게 연결):
```markdown
Gateway가 직접 인증·인가를 처리하지 않고 별도 서비스에 위임하는 구조입니다.
요청을 수신하고 판정을 받은 뒤 백엔드로 전달합니다.
```

Good (명시적 대조 어휘가 있어 쉼표 유지):
```markdown
선택한 것이 아니라, 주어진 환경에서 결정한 것입니다.
```

## PN3. 열거 접속어 앞 쉼표 금지

3개 이상 항목 열거에서 쉼표로 구분하되, 마지막 항목 앞 "등·및·그리고" 바로 앞에는 쉼표를 찍지 않는다.
이는 영문 Oxford comma 관습의 잘못된 이식이다.

Bad:
```markdown
인증, 인가, 감사, 및 로깅을 별도 서비스로 분리합니다.
```

Good:
```markdown
인증, 인가, 감사 및 로깅을 별도 서비스로 분리합니다.
인증, 인가, 감사, 로깅을 별도 서비스로 분리합니다.
```

## PN4. 콜론은 도입에만 사용

콜론(`:`)은 목록·정의·인용을 도입할 때만 쓴다.
부사절 연결을 쉼표 대신 콜론으로 대체하는 표기는 영문 문체 잔재로 한국어에서는 어색하다.

Bad:
```markdown
Forward Auth의 장점: 인가 로직 중앙화, 스트리밍 병목 해소.
```

Good:
```markdown
Forward Auth의 장점은 인가 로직 중앙화와 스트리밍 병목 해소입니다.

Forward Auth의 장점은 다음과 같습니다.
- 인가 로직 중앙화
- 스트리밍 병목 해소
```

## PN5. 세미콜론 금지

한국어 본문에서 세미콜론(`;`) 사용을 금지한다.
한국어에는 세미콜론의 전통이 없고, 문장 연결은 마침표나 접속사로 처리한다.
세미콜론은 코드 예시에서만 허용된다 (언어 문법의 일부).

Bad:
```markdown
Forward Auth는 인가 위임 방식입니다; 데이터 경로와 인가 경로를 분리합니다.
```

Good:
```markdown
Forward Auth는 인가 위임 방식입니다. 데이터 경로와 인가 경로를 분리합니다.
Forward Auth는 인가 위임 방식으로, 데이터 경로와 인가 경로를 분리합니다.
```

## PN6. 가운뎃점(·)으로 짧은 병렬 묶기 (슬래시 병렬 지양)

대등한 명사를 짧게 나열할 때는 가운뎃점(`·`)으로 묶는다.
같은 자리에 슬래시(`/`)를 쓰면 "둘 중 하나"나 "경로·분수"로 읽혀 병렬 의도가 흐려지고 가독성이 떨어진다.
가운뎃점은 국립국어원 문장 부호 규정에서 짝을 이루는 어구를 묶을 때 허용하는 부호다.

가운뎃점이 네 항목을 넘어 길어지면 쉼표나 목록으로 분리한다.

**슬래시 유지 예외** (병렬이 아닌 관용 표기):
- 기술 관용: `I/O`, `A/B 테스트`, `TCP/IP`, `km/h`
- 코드·경로·식별자: `user.name/email`, `Edit/Write`, `and/or`
- 분수·비율·날짜: `1/2`, `03/17`

Bad:
```markdown
구축/운영하는 팀이 함께 정합니다.
작성/검증/발행 흐름을 한 번에 봅니다.
```

Good:
```markdown
구축·운영하는 팀이 함께 정합니다.
작성·검증·발행 흐름을 한 번에 봅니다.
```
