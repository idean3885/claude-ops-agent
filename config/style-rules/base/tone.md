# 저자 톤 (base)

저자의 글이 기계적·구어체·즉흥적으로 읽히지 않게 유지하는 규칙.
`readability.md K2`(합쇼체 통일)를 전제로 하며, 그 위에 문장 구조·용어 선택·메타 구성의 일관성을 더한다.

> **연계**: 톤 위반은 대부분 `ai-tells.md`의 카테고리와 1:1 매핑된다. 예: T1 ↔ J-3 (em dash), T8 ↔ A-7·D-5 등.

---

## T1. em dash(—) 금지

한국어 본문에서 em dash(`—`) 사용을 금지한다.
**콜론(`:`)**, **쉼표(`,`)**, **괄호(`()`)**, **줄바꿈** 중 문맥에 맞는 기호로 대체한다.

> **근거**: em dash는 영어권 편집 관행에서 주로 쓰이며, 한국어 문어에서는 읽는 리듬이 끊긴다.
> AI가 생성한 한국어 글에서 과도하게 등장하여 "AI 티"의 대표 신호가 됨.

Bad:
```markdown
Forward Auth는 데이터 경로와 인가 경로의 분리 — 이것이 핵심이다.
```

Good:
```markdown
Forward Auth의 핵심은 데이터 경로와 인가 경로의 분리다.
Forward Auth의 핵심은 데이터 경로와 인가 경로의 분리(decoupling)다.
```

`ai-tells.md J-3` 와 동일 규칙.

## T2. 어순 전환·두 문장 도치 금지

구어체 목적어 종결(`~다는 걸요`, `~사실을요`)을 금지한다.
또한 "완결 선언 후 목적절만으로 된 뒷 문장"을 덧붙이는 두 문장 도치 구조도 금지한다.
목적어가 필요하면 한 문장 안에서 완결한다.

Bad:
```markdown
나중에야 알게 됐습니다. 이 방식이 정형 패턴이라는 걸요.
나중에야 알게 됐습니다. 이 방식이 정형 패턴이라는 것을.
```

Good:
```markdown
이 방식이 정형 패턴이라는 사실을 나중에야 알게 됐습니다.
```

## T3. 서술어 생략된 체언 종결 금지

"A는 종전과 동일, B는 새 필드." 처럼 서술어가 생략된 명사구 종결 문장을 금지한다.
대비·나열이 필요해도 각 절에 서술어를 붙여 완결한다.

Bad:
```markdown
외부에서 보면 기존 API는 종전과 동일, 신규 API는 새 필드.
```

Good:
```markdown
외부에서 보면 기존 API는 종전 응답을 그대로 유지하고,
신규 API만 새 필드를 노출합니다.
```

---

## T4. 도입부 메타 설명 중복 금지

도입부에 "이 글이 무엇을 다루는가"를 설명하는 메타 문단을 **1회로 제한**한다.
TL;DR이 이미 메타 요약 역할을 하므로, 본문 진입 전 별도 "이 글은 ~에 중심을 둡니다" 같은 문단을 추가하지 않는다.

> **근거**: 메타 설명이 연속되면 본론 진입이 지연되어 독자가 이탈한다.
> TL;DR + 도입부 한 문단이면 충분하다.

---

## T5. 즉석 명명 회피, 교과서 용어 우선

정식 용어(업계 표준, 교과서 표기)가 존재하는 개념을 "편의상 X라고 부르겠습니다" 식으로 즉석 명명하지 않는다.
먼저 정식 용어를 탐색하고, 기존 용어가 있으면 그것을 사용한다.
즉석 명명이 불가피한 경우(팀 내 전용 약어 등)에만 명시적 도입을 허용한다.

Bad:
```markdown
이렇게 두 버전이 공존해도 깨지지 않는 상태를 편의상 N-1 호환성이라고 부르겠습니다.
```

Good:
```markdown
이렇게 여러 버전이 공존하는 중간 상태를 **version skew**라고 부릅니다.
이 상태에서 깨지지 않는 성질이 **하위 호환성(backward compatibility)**입니다.
```

## T6. 영어 전문 용어 첫 등장 시 괄호 풀이

`readability.md K3`(외래어 표기) 원칙을 톤 차원에서 강화한다.
`write-both`, `version skew`, `backward compatibility`, `schema-only` 등 영어 전문 용어는
**첫 등장 시 괄호로 한국어 풀이 또는 한국어 표기 + 괄호 원어 병기**한다.
이후 반복은 원어만 써도 무방하다.

Bad:
```markdown
Migrate 단계에서 쓰기는 write-both 기간을 둡니다.
```

Good:
```markdown
Migrate 단계에서 쓰기는 레거시·신규 양쪽에 하는(write-both) 기간을 둡니다.
```

## T7. 기계적 수치 강조체 금지

숫자를 형용사처럼 붙여 강조하는 표현을 금지한다.
`변화 0`, `비용 0`, `위험 0`, `의존 0`, `차이 0` 같은 패턴은 영어식 간결체("zero impact", "no change")를 반사 번역한 결과로,
한국어 기술 문서 관용에 맞지 않아 공신력을 떨어뜨린다.

Bad:
```markdown
외부 계약 변화 0이라 독립 MINOR bump는 불필요합니다.
이 경로는 런타임 의존 0, 빌드 비용 0입니다.
```

Good:
```markdown
외부 계약에 변화가 없어 독립 MINOR bump는 불필요합니다.
이 경로는 런타임 의존이 없고, 빌드 비용도 들지 않습니다.
```

## T8. 구어체·감정체 동사 금지

두 범주의 동사를 기술 문서 본문에서 사용하지 않는다.

1. **감정·파괴 계열**: `터지다`, `박살나다`, `날아가다`, `죽다`
   → 중립 동사(`발생하다`, `손상되다`, `유실되다`, `중단되다`, `실패하다`)로 대체
2. **물리 조작 계열**: `떼어내다`, `끼우다`, `박아넣다`, `쑤셔넣다`, `꽂다`, `묶다`(제한·구속 의미로 쓸 때), `빼다`(제거 의미)
   → 추상적 중립 동사(`분리하다`, `삽입하다`, `포함하다`, `개입하다`, `제한하다`, `제외하다`)로 대체

합쇼체 포멀 톤과 충돌하고, 사실 서술을 감정·행위 서술로 바꿔 공신력을 떨어뜨린다.

> **연계**: `forbidden-words.json`의 `박았|박힘|박혀|박힌|박은|박아|박혔` 패턴은 이 규칙의 hook 실행 layer.

Bad (감정·파괴):
```markdown
문제가 터졌을 때 돌아갈 수 없음.
배포가 실패하면 데이터가 날아갑니다.
장애 나면 서비스가 죽습니다.
```

Good:
```markdown
문제가 발생했을 때 돌아갈 수 없음.
배포가 실패하면 데이터가 유실됩니다.
장애 시 서비스가 중단됩니다.
```

Bad (물리 조작):
```markdown
인가 서비스를 데이터 경로에서 떼어내는 방법을 찾았습니다.
인가 서비스가 데이터 경로에 끼어 대역폭을 묶는 문제가 있습니다.
경로에서 불필요한 인가 경유를 빼고 판정만 남깁니다.
```

Good:
```markdown
인가 서비스를 데이터 경로에서 분리하는 방법을 찾았습니다.
인가 서비스가 데이터 경로에 개입해 대역폭을 제한하는 문제가 있습니다.
경로에서 불필요한 인가 경유를 제외하고 판정만 남깁니다.
```

## T9. 임의 신조어 합성 금지

기술 약어·태그(`chore`, `feat`, `fix`, `WIP`, `PR`, `MR` 등) 뒤에
문학적·구어적 동사 어근을 붙여 **즉석 합성어**를 만들지 않는다.

> **근거**: 기술 약어는 관습적으로 고정된 범주다.
> `chore 편승`, `feat 끼워넣기`, `WIP 얹기` 같은 합성은 저자가 만든 일회성 표현으로,
> 독자에게 친숙하지 않고 번역·검색 모두에서 통용되지 않는다.

Bad:
```markdown
SemVer 관점: chore 편승
이번 작업은 다음 릴리즈에 feat 끼워넣기 예정입니다.
```

Good:
```markdown
SemVer 관점: 독립 bump 불필요
이번 작업은 다음 릴리즈에 chore 커밋으로 함께 포함시킬 예정입니다.
```

## T10. 키보드 외 수학·논리 기호 남용 금지

수학·논리 기호(`⊆`, `⊂`, `⊇`, `⊃`, `∈`, `∉`, `∀`, `∃`, `∅`, `≤`, `≥`, `≠`, `≈`, `⇒`, `⇔`, `∴`, `∵`)를 기술 문서 본문에서 사용하지 않는다.
키보드로 직접 입력할 수 없어 저자가 매번 복사·유니코드 입력을 해야 하고,
독자도 동일한 이유로 검색·인용이 어렵다.

- 자연어로 풀어 쓴다: `부분집합`, `이상·이하`, `같지 않다`, `따라서`, `왜냐하면`
- 코드·수식 맥락에서는 ASCII로 대체한다: `!=`, `<=`, `>=`, `->`, `=>`

Bad:
```markdown
"파티션 키 ⊆ 유일성 키" 제약 때문에 단일 PK를 포기했습니다.
@A ≠ @B 라는 점에 주의해야 합니다.
x ∈ S 인 모든 원소에 대해 검증합니다.
```

Good:
```markdown
"파티션 키는 유일성 키의 부분집합"이라는 제약 때문에 단일 PK를 포기했습니다.
@A와 @B는 다르다는 점(@A != @B)에 주의해야 합니다.
집합 S에 속하는 모든 원소에 대해 검증합니다.
```

`ai-tells.md J-4` 와 동일 규칙.

## T11. 권위 용어 남발 금지

"패턴", "표준", "베스트 프랙티스", "모범 사례" 같은 권위 용어는
**공신력 있는 참조가 있을 때만** 사용한다.
통용 용어·구현체별로 이름이 분화된 개념·저자 개인 명명에는
"방식", "접근법", "구조" 같은 중립 명사를 쓴다.

**허용되는 공신력 참조 예시**:
- GoF 디자인 패턴: Singleton, Factory, Observer 등
- Enterprise Integration Patterns, Martin Fowler 저서 수록 패턴: Circuit Breaker, Strangler Fig, BFF, Saga, CQRS 등
- 공식 표준·RFC: HTTP, OAuth 2.0, OIDC 등

Bad:
```markdown
Forward Auth 패턴: GW가 판정을 위임하는 패턴입니다.
(Forward Auth는 구현체별로 이름이 분화된 통용 용어)

REST 패턴을 따른 것이 베스트 프랙티스입니다.
(개인 판단에 "베스트 프랙티스" 붙임)
```

Good:
```markdown
Forward Auth는 GW가 판정을 위임하는 방식입니다.

여기서는 REST 스타일을 따르는 것이 적합하다고 판단했습니다.
```

## T12. 분석 틀 라벨 헤딩 노출 금지

PAAR·STAR·ABCDE 같은 분석 프레임워크의 단계 표시(`P.`, `A1.`, `R.`),
순차 단계 번호(`Step 1:`, `Phase 2:`), 알파벳 분류 표지(`A.`, `B.`, `C.`)를
헤딩에 노출하지 않는다. 구조는 콘텐츠 흐름과 헤딩 텍스트에서 자연스럽게 드러나야 한다.

Bad:
```markdown
## P. 문제
## A1. 어디서 stale이 발생하는가
## A2. 근본 원인
## A3. 재검증 흐름
## A4. 수정
## R. 결과와 재실측
```

Good:
```markdown
## 문제
## 어디서 stale이 발생하는가
## 근본 원인
## 재검증 흐름
## 수정
## 결과와 재실측
```

자동 판별: 헤딩 텍스트가 `^[A-Z][0-9]?\.\s` 또는 `^Step \d+:` 또는 `^Phase \d+:` 패턴으로 시작하는가.

## T13. 사례·관찰 서술은 1인칭 회상으로 통일

저자가 직접 관여한 작업·관찰·판단을 묘사할 때는 1인칭 회상 톤(`확인했습니다`, `예상했습니다`, `추적했습니다`, `재현했습니다`)을 쓴다.
3인칭 기계식 묘사나 익명 수동태(`재현되었습니다`, `검출되지 않았습니다`)는 회피한다.

> **근거**: 글을 쓰는 시점에 저자는 이미 1인칭이다.
> 사례 회상을 시스템 보고서처럼 3인칭 기계식으로 묘사하면 저자가 사라진 보고서 톤이 되어 회고 서술의 저자성·신뢰도가 떨어진다.

**예외 (3인칭 현재형 허용)**:
- 시스템·표준의 일반 동작 정의: `브라우저는 같은 URL 재방문 시 ... 안 합니다`, `Swift는 PUT마다 ETag를 발급합니다`
- 인용·참조의 객관 진술

Bad:
```markdown
4번이 적용되어 브라우저가 자체 추정으로 진입했습니다.
테스트 환경에서 e2e로 재현되었습니다.
ETag 변경은 검출되지 않았습니다.
```

Good:
```markdown
1~3번이 빠진 상태라 4번 heuristic이 적용된 것이라고 예상하고 추적했습니다.
테스트 환경에서 e2e로 재현했습니다.
ETag 변경을 알아챌 기회 자체가 없었습니다.
```
