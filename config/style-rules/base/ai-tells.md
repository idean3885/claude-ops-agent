# AI 티 분류 SSOT (base)

한국어 글(블로그·위키·이슈·PoC·데일리로그·동료리뷰 등 모든 문서)에서
AI 가 쓴 글의 흔적("AI 티")을 식별·차단·교정하기 위한 분류 체계.

> **출처**: 분류 체계 골격(A~J 10대 카테고리, 심각도 S1/S2/S3)은
> [`epoko77-ai/im-not-ai`](https://github.com/epoko77-ai/im-not-ai) (MIT) 의 한국어 humanize 스킬에서 차용했다.
> 본 SSOT 는 그 골격 위에 한국어 기술 블로그·사내 위키·PoC 문서 맥락의 처방·예시·forbidden-words hook 패턴 매핑을 자체 작성한 파생물이다.
> 라이선스: im-not-ai 의 MIT 조항을 준수하며, 본 SSOT 도 동일 MIT 로 배포된다.

---

## 적용 범위

이 SSOT 는 **모든 한국어 문서**에 적용되는 기초 룰이다.
문서 유형별 추가 규약은 `../extensions/{blog,wiki,poc,info,issue,dailylog,peer-review}.md` 에서 base 를 import 한 뒤 얹는다.

소비자:
- `devex/hooks/forbidden-words-{prompt,stop}.sh` — `base/forbidden-words.json` 의 S1 패턴을 응답 출력 단계에서 즉시 차단·경고
- `devex:content-write` / `devex:content-verify` / `devex:content-publish` — 작성·검증·발행 시 base + extension 로드
- `toolkit:content-write` / `toolkit:wiki-publish` — 사내 위키 어댑터. `~/.claude/devex/style-rules/` 미러 경로를 참조

## 4대 철칙

검증·윤문이 의미를 훼손하지 않도록 모든 처방에 우선한다.

1. **의미 불변** — 사실·주장·수치·고유명사·직접 인용은 100% 원문 보존
2. **근거 기반** — 탐지된 span 에만 수술적 수정. 탐지 없는 구간은 건드리지 않음
3. **장르 유지** — 칼럼을 문학으로, 리포트를 에세이로 옮기지 않음
4. **과윤문 금지** — 변경률 30% 초과 시 경고, 50% 초과 시 강제 중단

## 심각도

| 등급 | 정의 | 처리 |
|------|------|------|
| **S1** | 한 번만 등장해도 AI 확신을 주는 결정적 신호 | 무조건 제거. `forbidden-words.json` 에 hook 패턴으로 등록 |
| **S2** | 1~2회는 허용, 3회 이상 반복 시 제거 | 검증 단계에서 빈도 카운트 후 결정 |
| **S3** | 단독으로는 약함. 다른 S1/S2 와 중첩될 때만 문제 | 다른 신호의 가중치로만 사용 |

윤문/검증 완료 후 부여한다. extension 별 합격선은 다를 수 있다.

| 등급 | 기준 |
|------|------|
| **A** | S1 0건, S2 ≤2건, 개선률 70% 이상 |
| **B** | S1 0건, S2 ≤4건, 개선률 50% 이상 |
| **C** | S1 1~2건 또는 과윤문 신호 2개 → 2차 윤문 필요 |
| **D** | S1 3건 이상 또는 심각한 과윤문 → 사람 검토 권고 |

## Do-NOT 리스트 (탐지·교정 제외)

다음은 패턴이 매칭되어도 건드리지 않는다.

- 수치·단위·날짜
- 고유명사·인명·제품명·모델명·라이브러리명
- 큰따옴표 내부 직접 인용
- 코드 블록(``` ``` ```), 인라인 코드(`` ` ``)
- 법률·규정·표준 문서 조문 인용
- 학술 개념어가 불가피한 경우 (예: "타동성", "관계절 좌향 수식" 같은 메타 용어)

## 분류 체계 (A~J)

### A. 번역투 (영어 직역체)

가장 흔하고 결정적인 AI 티. 영어 원문 어순·관용 표현을 한국어로 직역한 결과.

| ID | 패턴 | 심각도 | 처방 | hook 매핑 |
|----|------|-------|------|----------|
| A-1 | `~을 통해`, `~를 통해` | **S1** | `~로`, `~으로` (직접 수단) | `을 통해|를 통해` |
| A-2 | `~에 있어서`, `~에 있어` | S2 | `~에서`, `~의 경우`, 또는 삭제 | - |
| A-3 | `~에 대해`, `~에 관해` | S2 | 직접 목적격 (`~를`, `~에 관한`) | - |
| A-4 | 이중 피동 (`~되어지다`, `~지어지다`) | **S1** | 단순 피동 (`~되다`) 또는 능동 | - |
| A-5 | `~에 의해 ~된` | S2 | 능동 (`~가 ~한`) | - |
| A-6 | `그/그녀/그것/그들` 강박적 매핑 | S2 | 한국어는 주어 생략 가능. 반복 호명 제거 | - |
| A-7 | Light verb 분리 구문 (`have/make/take/give + 명사`) → `~을 가지다`, `~을 만들다` | S2 | 단일 동사 (`있다`, `만들다`, `결정하다`) | - |
| A-8 | 관계절 좌향 수식 (관형구 3중 중첩) | S2 | 문장 분할 또는 우향 수식 | - |
| A-9 | 이중 조사 (`~에서의`, `~에로의`, `~으로의`, `~에의`) | S2 | 단순 조사 (`~의`, `~에서`, `~로`) | - |
| A-10 | 사역·인지·발화 동사 분리 (`~하도록 하다`, `~게 하다`) | S2 | 직접 사역 (`~시키다`) 또는 능동 | - |
| A-11 | `활용하여`, `활용한`, `활용함으로써` | **S1** | `써서`, `사용하여` | `활용하여|활용함으로써|활용한` |
| A-12 | `본 (문서|시스템|기능|보고서|작업)은` | **S1** | 주어 직접 명시 (`이 문서는`, `이 시스템은`) | `본 (문서|시스템|기능|보고서|작업)은` |

**Bad** (A-1, A-8):
```
AI 기술을 통해 효율을 높일 수 있는 다양한 기능을 제공하는 시스템에 대해 설명합니다.
```

**Good**:
```
이 시스템은 AI 로 효율을 높이는 기능을 제공합니다.
```

### B. 영어 인용·용어 과다

번역 가능한 영어를 한국어 옆에 병기하거나 영어 그대로 두는 패턴.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| B-1 | 한국어 옆 영어 괄호 병기 남발 (`효율(efficiency)`, `사용자(user)`) | S2 | 첫 등장 1회만 병기. 이후 한국어만 |
| B-2 | 번역 가능한 영어 그대로 (`provide`, `deliver`, `implement`) | S2 | 한국어 동사 (`제공하다`, `구현하다`) |
| B-3 | 약어 풀이 없음 (`PR`, `CI/CD`, `WAS` 등) | S3 | 첫 등장 시 풀이. 단 업계 표준 약어는 예외 |

`readability.md K3` (외래어 표기) 와 연계. 기술 용어 첫 등장 시 `한국어(English)` 형식.

### C. 구조적 AI 패턴

문장이 아니라 문서 구조에서 드러나는 AI 티.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| C-1 | 기계적 `첫째/둘째/셋째` 병렬 | S2 | 문맥에 맞는 접속(`먼저`, `그다음`, `마지막으로`) 또는 목록 |
| C-2 | 과한 불릿 (모든 문장이 불릿) | S2 | 산문 5할 이상 회복 |
| C-3 | 과한 헤딩 (H2/H3 가 3문장마다 등장) | S2 | 헤딩 1개 = 2~4 문단 분량 |
| C-4 | 이모지 남발 (✅ ❌ 🚀 ⚡ 🎯 등) | **S1** | 본문 이모지 제거. 표 상태 표시 등 기능적 용도만 허용 |
| C-5 | 연결어미 뒤 쉼표 (`~하고,`, `~지만,`) | **S1** | 쉼표 제거 또는 문장 분할. `readability.md PN2` 와 연계 |
| C-6 | 쉼표 포함률 과다 (전체 문장 중 50%+ 가 쉼표 보유) | S2 | 호흡 장식 쉼표 제거 |

**Bad** (C-4, C-2):
```
## ✅ 장점
- 🚀 빠른 속도
- ⚡ 낮은 메모리
- 🎯 명확한 API
```

**Good**:
```
## 장점

속도가 빠르고 메모리 사용량이 적으며 API 가 명확합니다.
```

### D. AI 특유 관용구

AI 모델이 결론·도입·강조에 반복적으로 동원하는 어구.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| D-1 | `결론적으로`, `요컨대`, `종합하자면` | **S1** | 결론은 직접 진술. 도입어 삭제 |
| D-2 | `시사하는 바가 크다`, `의미가 깊다` | **S1** | 무엇을 시사하는지 직접 기술 |
| D-3 | `주목할 만하다`, `눈여겨볼 만하다` | **S1** | 평가 대신 사실 기술 |
| D-4 | `혁신적인`, `획기적인`, `차세대` | **S1** | 구체 속성 (어떤 점에서) |
| D-5 | `다양한` 단독 사용 | S2 | 구체 항목 또는 수 (`3가지`, `여러 종류의`) |
| D-6 | `포괄적인`, `체계적인`, `효율적인`, `총체적인`, `유기적인` | **S1** | 구체 표현 (무엇을/어떻게) | `포괄적|체계적|효율적|총체적|일관된 흐름|유기적` |
| D-7 | `강력한`, `우아한`, `견고한`, `튼튼한 구조` | **S1** | 구체 속성 (어떤 점에서) | `강력한|우아한|견고한|튼튼한 구조` |
| D-8 | `원활(한|히|하게|함)`, `매끄(럽|러운|러이)` | **S1** | 구체 동작 동사 | `원활(한|히|하게|함)|매끄(럽|러운|러이)` |
| D-9 | `심층(적|적인|있는)`, `심도(있는|깊은)` | **S1** | 범위 구체 명시 | `심층(적|적인|있는)|심도(있는|깊은)` |
| D-10 | `보장(합니다|됩니다|된)`, `담보(합니다|됩니다)` | **S1** | 조건·근거 직접 기술 | `보장(합니다|됩니다|된)|담보(합니다|됩니다)` |
| D-11 | `고수준`, `저수준`, `고차원`, `저차원적` (수식어로 쓸 때) | **S1** | 기능을 직접 기술 | `고수준|저수준|고차원|저차원적` |

D-6~D-11 은 모두 `forbidden-words.json` 에 hook 패턴으로 등록되어 응답 출력 단계에서 즉시 차단된다.

### E. 리듬 균일성

문장 호흡이 평탄해서 사람이 쓴 글의 자연스러운 변주가 사라진 상태.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| E-1 | 문장 길이 표준편차 < 5 (전체 문장이 비슷한 길이) | S2 | 짧은 문장·긴 문장 의도적 혼용 |
| E-2 | 동일 종결어미 5문장 연속 (`~합니다.` x 5+) | S2 | 종결어미 변주 (서술 / 의문 / 명령) — 단 합쇼체 유지 |
| E-3 | 진행형 `~하고 있다` 자동 매핑 | S3 | 현재형 (`~하다`) 단순화. 진행 의미가 중요할 때만 유지 |
| E-4 | 청자 경어법 손실 (반말↔합쇼체 혼용) | S2 | `readability.md K2` (합쇼체 통일) 적용 |

### F. 수식·중복

형용사·부사·접미사를 반복적으로 덧붙이는 패턴.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| F-1 | `매우`, `정말`, `상당히`, `굉장히` 빈출 | S2 | 정량 (`3배`, `0.5초`) 또는 삭제 |
| F-2 | 동의어 이중 수식 (`빠르고 신속한`, `중요하고 핵심적인`) | S2 | 둘 중 하나만 |
| F-3 | 한자어 명사화 접미사 `~적`, `~성`, `~화` 남발 | S2 | 구체 동사·명사 |
| F-4 | 영어 명사화 접미사 직역 (`-tion`, `-ment`, `-ness`, `-ity`) | S2 | 한국어 동사 또는 명사 |

### G. Hedging 남용

자신감 부족이 아니라 과한 완곡으로 책임 회피적 인상을 주는 표현.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| G-1 | `~할 수 있을 것으로 보인다`, `~할 가능성이 있다고 판단된다` (다중 완곡) | S2 | 단일 hedge (`~로 보인다`) 또는 단언 |
| G-2 | 안전 균형 lexicon (`물론`, `그러나`, `하지만`, `다만`, `한편` 빈출) | S2 | 회수 조절 |

### H. 접속사 남발

문두 접속사가 연속 등장해 글의 기계적 짜임새가 드러나는 상태.

| ID | 패턴 | 심각도 | 처방 |
|----|------|-------|------|
| H-1 | 문두 `또한` 3회 이상 등장 | **S1** | 첫 등장만 유지. 나머지 삭제 |
| H-2 | 문두 `따라서`, `그러므로`, `결국` 연속 | S2 | 인과는 문장 안에 녹여 표현 |
| H-3 | 문두 `즉`, `다시 말해`, `나아가` 연속 | S2 | 반복 회수 조절 |

|----|------|-------|------|

| I-3 | `~할 필요가 있다` | **S1** | `~해야 한다`, `~한다` |
| I-4 | `~다는 것`, `~라는 것` | S2 | 명사절 단순화 |

### J. 시각 장식 남용

마크다운 시각 요소(볼드, 따옴표, 대시 등) 과사용.

### K. 감정체·의인화 표현

사람 상태 형용사로 시스템을 묘사하거나 시스템에 생명·감정을 부여하는 패턴. 기술 문서 톤을 흐리고 도메인 용어 일관성을 깨뜨린다.

| ID | 패턴 | 심각도 | 처방 | hook 매핑 |
|----|------|-------|------|----------|
| K-1 | `멀쩡(한\|하다\|히\|함)` (사람 상태 형용사로 시스템·서비스 묘사) | **S1** | 도메인 동사 (`중단되지 않는다`, `유지된다`, `동작한다`) | `멀쩡(한\|하다\|히\|함)` |
| K-2 | 시스템·프로세스·서비스 의인화 동사 (`죽다`, `살아남다`, `뻗다`, `쓰러지다`) | S2 | 도메인 동사 (`다운된다`, `동작 중이다`, `정지한다`). TCP `연결이 끊기다` 같은 IT 표준 한국어는 예외 |
| K-3 | 구어 감정체 (`박살`, `쌩쌩`, `먹통` 등 — 시스템 묘사 맥락) | S2 | 도메인 동사 |

**Bad** (K-1, K-2):
```
컨트롤 플레인이 죽어도 사용자 파드는 멀쩡합니다.
```

**Good**:
```
컨트롤 플레인이 다운되어도 사용자 파드는 중단되지 않습니다.
```

**도메인 용어 일관성**: 본문에서 `무중단` / `중단` 같은 어휘 쌍을 일관되게 쓴다. `끊김`·`멀쩡`·`죽음`을 섞으면 어휘 쌍 정합이 깨져 톤이 흐려진다.

## forbidden-words.json 매핑 표

`base/forbidden-words.json` 의 10개 룰은 위 분류 체계의 S1 패턴 중 응답 출력 단계에서 즉시 차단할 가치가 있는 것들이다. 카테고리 ID 대응:

| forbidden-words 패턴 | 카테고리 ID | reason |
|--------------------|----------|--------|
| `박았\|박힘\|박혀\|박힌\|박은\|박아\|박혔` | tone.md T8 (물리 조작 동사) | 비격식·저속 |
| `고수준\|저수준\|고차원\|저차원적` | D-11 | 과시성 수식어 |
| `포괄적\|체계적\|효율적\|총체적\|일관된 흐름\|유기적` | D-6 | 추상적 과장 형용사 |
| `원활(한\|히\|하게\|함)\|매끄(럽\|러운\|러이)` | D-8 | 의미 없는 부사·형용사 |
| `강력한\|우아한\|견고한\|튼튼한 구조` | D-7 | 과장형 형용사 |
| `을 통해\|를 통해` | A-1 | 영어 'through' 직역체 |
| `활용하여\|활용함으로써\|활용한` | A-11 | 격식 과잉 |
| `본 (문서\|시스템\|기능\|보고서\|작업)은` | A-12 | 보고서체 |
| `보장(합니다\|됩니다\|된)\|담보(합니다\|됩니다)` | D-10 | 근거 없는 단언 |
| `심층(적\|적인\|있는)\|심도(있는\|깊은)` | D-9 | 깊이 자랑 |
| `멀쩡(한\|하다\|히\|함)` | K-1 | 감정체·의인화 (사람 상태 형용사로 시스템 묘사) |

신규 패턴은 분류 체계에 카테고리 ID 부여 후 S1 으로 판정될 때 `forbidden-words.json` 에 추가한다.
사용자 추가 룰은 `~/.claude/forbidden-words.local.json` 에 작성 (devex hook 이 머지).

## 검증 절차 (요약)

상세는 `devex:content-verify` SKILL.md 와 `readability.md` 의 체크리스트 표를 참조한다. base 단계에서는 다음 순서를 따른다.

1. **카테고리 매칭**: 본문에서 A~J 패턴 탐지 (span 단위)
2. **심각도 분류**: 탐지된 span 을 S1/S2/S3 로 분류
3. **카운트 집계**: S1 건수, S2 빈도
4. **품질 등급 판정**: A/B/C/D
5. **수술적 윤문**: 탐지된 span 만 처방대로 교정 (변경률 모니터링)
6. **재검증**: 잔존 패턴 + 과윤문 신호 재측정

```

extension 은 base 를 대체하지 않는다. **추가**만 한다.
