# 블로그 익스텐션

> **Base**: `base/readability.md` + `base/tone.md` + `base/punctuation.md` + `base/ai-tells.md`
> **Adds**: 블로그 포스팅 특화 규칙 (포트폴리오 어필, PAAR 구조, 카테고리·태그 메타)

블로그 포스팅은 저자의 의사결정·실무 경험을 외부 독자(면접관 포함)에게 전달하는 매체다.
사내 위키 원본을 발행하는 흐름에서는 `wiki.md` 익스텐션을 거친 뒤 블로그용으로 가공한다.

---

## 적용 대상

| 매체 | 적용 |
|------|------|
| Jekyll Chirpy 블로그 (`idean3885.github.io`) | 본 익스텐션 적용 |
| 외부 기술 블로그 (Medium·Velog·tistory 등) | 본 익스텐션 + 매체별 메타 가공 |
| 사내 위키 원본 | `wiki.md` 익스텐션 (블로그 발행 전 단계) |

## B1. PAAR 구조 필수 (프로젝트 회고·경험형)

저자 고유 경험·의사결정 근거가 핵심인 글은 PAAR 구조를 따른다.

- **문제 (Problem)**: 왜 이 문제가 중요했는가
- **분석 (Analyze)**: 어떤 대안을 검토했고, 왜 이 방법을 선택했는가 (**핵심 섹션**)
- **해결 (Action)**: 무엇을 구현했는가
- **결과 (Result)**: 수치 기반 성과

> AI 시대에 시리즈 양(편수)은 차별화 요소가 아닙니다.
> AI가 생성할 수 없는 것은 저자의 판단과 트레이드오프 분석뿐입니다.
> 시리즈 5편보다 PAAR이 제대로 된 1편이 더 유의미합니다.

PAAR 라벨을 헤딩에 노출하는 것은 `tone.md T12` 위반이다. 라벨 없이 흐름만 따른다.

## B2. TL;DR 필수 (1500자 이상)

`readability.md V3`의 권장을 블로그에서는 **필수**로 격상한다.

```markdown
## TL;DR

- 핵심 결론
- 핵심 수치
- 적용 범위·한계
```

## B3. 시리즈 인덱스 = 포트폴리오 랜딩

시리즈의 상위 페이지(인덱스)는 면접관·외부 독자의 1분 내 파악용 랜딩이다.

| 역할 | 인덱스 (블로그) |
|------|---------------|
| 목적 | 이력서·포트폴리오 랜딩 페이지 |
| 내용 | 프로젝트 요약 + 각 편 핵심 + 링크 |
| 독자 | 면접관 (1분 내 전체 파악용) |
| 갱신 | 시리즈 현행화 시 합본 역할 |

---

## B4. 개별 편 독립성

시리즈 각 편은 단독으로 읽혀야 한다.

| 원칙 | 설명 |
|------|------|
| 독립 도입부 | 시리즈 순서 상관없이 이해 가능한 배경 설명 |
| 독립 제목 | "N편" 번호보다 주제가 먼저 드러나는 제목 |
| 독립 가치 | 각 편이 하나의 문제→해결 스토리를 완결 |

독립 가치가 낮은 편은 보강하거나 다른 편에 흡수를 검토한다.

---

## B5. 꾸준한 출력 시그널

블로그의 포스팅 수와 날짜 분포는 "꾸준히 쓰는 사람"이라는 시그널이다.
단편 베이스(B7)는 이 시그널과 직접 맞는다: 깊이 있는 단편이 일정 간격으로 쌓인다.
시리즈로 묶을 때는 각 편의 독립 가치를 확보해 빈도 손실을 막는다.

---

## B6. 단편 우선 (시리즈는 예외)

포스팅 기본 단위는 단편이다. 시리즈는 한 편으로 담기 어려운 연속 서사에서만 택한다.

| 구분 | 단편 (기본) | 시리즈 (예외) |
|------|-------------|---------------|
| 적용 | 대부분의 글 | 한 편으로 담기 어려운 연속 서사 |
| 판단 기준 | 하나의 문제→해결이 완결되는가 | 단편으로 쪼개면 각 편이 얕아지는가 |
| 깊이 | 단편 하나에 깊은 고찰을 담는다 | 분산 시 사상이 흩어지지 않는가 |

분할·시리즈를 택하기 전 점검한다: 단편 1편으로 압축 가능한가. 가능하면 단편으로 쓴다.
얕은 글을 시리즈로 늘리면 약한 편이 글 전체의 신뢰도를 떨어뜨린다.

## B7. 드래프트 단계

작성 직후 곧바로 발행하지 않는다. 드래프트 상태로 두고 점검한 뒤 발행한다.

| 점검 항목 | 통과 기준 |
|----------|-----------|
| PAAR Analyze 깊이 | 단편 자립이 가능한 수준의 대안 검토·트레이드오프 |
| 저자 사상 | 판단·근거가 드러나는가 |
| 대체 불가성 | 정의·절차 나열이 아닌 저자 고유 경험인가 |

점검 부족 시 보강하거나 발행을 보류한다. 발행은 점검 통과 후 진행한다.

## B9. 본문 어필 표현 금지 (`base/tone.md T15` 블로그 강화)

블로그는 평가자·외부 독자를 염두에 둔 매체다(B3). 그 목적은 매체를 고르고 주제를 정하는 **전략**이지, 독자에게 읽히는 **본문 문장**이 아니다.
본문 프로즈에 어필 목적을 직접 명시하는 순간, 독자는 판단이 아니라 의도를 먼저 읽고 신뢰를 거둔다. 강점은 대안 검토·트레이드오프·근거의 질로 드러낸다.

| 구분 | 위치 | 어필 표현 |
|------|------|----------|
| 매체 목적 (허용) | 익스텐션·발행 전략, 카테고리 설계 | "평가자 1분 파악용 랜딩" 같은 운영 메모 |
| 본문 프로즈 (금지) | 독자에게 게시되는 문장 | "면접 빈출", "시니어로 가는 발판", "포트폴리오로" |

대상 어휘(본문 서술문 등장 시 위반): `면접`, `채용`, `시니어`/`주니어`, `포트폴리오`, `이력서`, `역량을 보여`, `나를 증명`, `경쟁력`.

Bad:
```markdown
면접 빈출 주제 두 가지는 별도 글로 분리했습니다.
여러 각도에서 본 시야가 시니어로 가는 발판이라고 봅니다.
```

Good:
```markdown
더 깊게 다룬 두 주제는 별도 글로 분리했습니다.
한 각도에 매몰되지 않으려 여러 제약을 함께 놓고 판단했습니다.
```

> 이 어휘는 이슈·위키 등 다른 문서에서는 정당할 수 있어 글로벌 `forbidden-words.json` 에는 넣지 않는다.
> 블로그·평가자 지향 매체의 본문 검증에서만 적용한다.

## B10. 분할 단위 = 독자의 문제 (프로젝트가 아니라)

글을 몇 편으로 나눌지는 작업·프로젝트 단위가 아니라 **독자의 문제(검색 의도)** 단위로 정한다. 한 작업에서 서로 다른 검색어로 도착하는 다른 문제가 여럿 나오면, 각 문제를 **번호 없는 독립 단독 글**로 쓴다. 이는 시리즈(B6)가 아니라 별개 글이며, 발행 시점도 서로 독립이다.

강한 글일수록 프로젝트 프레임을 뒤로 두고 문제를 앞세운다. 글의 주어는 "이 프로젝트에서 무엇을 했다"가 아니라 "이 문제를 어떻게 풀었다"이다. 이 방향은 공개 매체에서 사내 맥락 노출도 줄인다.

**분할 품질 게이트** — 세 조건을 모두 통과할 때만 별도 글로 분리한다. 하나라도 통과하지 못하면 별도 글이 아니라 한 글의 섹션 또는 지식 문서 항목이다.

| 조건 | 통과 기준 |
|------|-----------|
| 독자 분기 | 다른 독자가 다른 검색어로 도착하는가 |
| 단독 성립 | 다른 글 없이 이 글만으로 성립하는가 |
| 서사 완결 | 문제→분석→선택→트레이드오프가 자체 완결되는가 |

"관점이 다르다"만으로 분리하지 않는다. 느슨하게 적용하면 과분할·분량 채우기로 약한 글이 늘어나 B6의 실패와 같아진다.

세 형태 구분:

| 형태 | 판정 | 설명 |
|------|------|------|
| 통짜 1편 | 지양 | 서로 다른 문제를 한 글에 함께 담아 초점이 흩어진다 |
| 얕은 시리즈 | 지양 (B6) | 한 문제를 순차 분할해 각 편이 얕아진다 |
| 축별 독립 단독 글 | 권장 | 다른 문제를 각각 완결한 별개 글. 번호 없이 독립 |

각 글의 길이는 그 문제에 필요한 개념만 본문에 담아 줄인다. 범용 개념 소개 같은 일반 배경은 별도 지식 문서(위키 등)에 두고 링크로 축약한다.

## 필수 구조

```markdown
## TL;DR (B2)
## 배경 (B1 Problem — 왜 이 문제가 중요했는가)

## 결과 (B1 Result — 수치 기반 성과)
## 회고 (선택 — 저자의 판단·배운 점·다음 편 예고)
```
