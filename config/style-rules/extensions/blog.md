# 블로그 익스텐션

> **Base**: `base/readability.md` + `base/tone.md` + `base/punctuation.md` + `base/ai-tells.md`
> **Adds**: 블로그 포스팅 특화 규칙 (포트폴리오 어필, PAAR 구조, 카테고리·태그 메타)

블로그 포스팅은 저자의 의사결정·실무 경험을 외부 독자(면접관 포함)에게 전달하는 매체다.
사내 위키 원본을 발행하는 흐름에서는 `wiki.md` 익스텐션을 거친 뒤 블로그용으로 가공한다.

| 매체 | 적용 |

| 사내 위키 원본 | `wiki.md` 익스텐션 (블로그 발행 전 단계) |

## B1. PAAR 구조 필수 (시리즈 어필·프로젝트 회고형)

저자 고유 경험·의사결정 근거가 핵심인 글은 PAAR 구조를 따른다.

- **문제 (Problem)**: 왜 이 문제가 중요했는가
- **분석 (Analyze)**: 어떤 대안을 검토했고, 왜 이 방법을 선택했는가 (**핵심 섹션**)
- **해결 (Action)**: 무엇을 구현했는가
- **결과 (Result)**: 수치 기반 성과

> AI 시대에 시리즈 양(편수)은 차별화 요소가 아닙니다.
> AI가 생성할 수 없는 것은 저자의 판단과 트레이드오프 분석뿐입니다.
> 시리즈 5편보다 PAAR이 제대로 된 1편이 더 유의미합니다.

PAAR 라벨을 헤딩에 노출하는 것은 `tone.md T12` 위반이다. 라벨 없이 흐름만 따른다.

## B2. TL;DR 필수 (1500자 이상)

`readability.md V3`의 권장을 블로그에서는 **필수**로 격상한다.

```markdown
## TL;DR

- 핵심 결론
- 핵심 수치
- 적용 범위·한계
```

## B3. 시리즈 인덱스 = 포트폴리오 랜딩

시리즈의 상위 페이지(인덱스)는 면접관·외부 독자의 1분 내 파악용 랜딩이다.

| 역할 | 인덱스 (블로그) |
|------|---------------|
| 목적 | 이력서·포트폴리오 랜딩 페이지 |
| 내용 | 프로젝트 요약 + 각 편 핵심 + 링크 |
| 독자 | 면접관 (1분 내 전체 파악용) |
| 갱신 | 시리즈 현행화 시 합본 역할 |

---

## B4. 개별 편 독립성

시리즈 각 편은 단독으로 읽혀야 한다.

| 원칙 | 설명 |
|------|------|
| 독립 도입부 | 시리즈 순서 상관없이 이해 가능한 배경 설명 |
| 독립 제목 | "N편" 번호보다 주제가 먼저 드러나는 제목 |
| 독립 가치 | 각 편이 하나의 문제→해결 스토리를 완결 |

독립 가치가 낮은 편은 보강하거나 다른 편에 흡수를 검토한다.

---

## B5. 꾸준한 출력 시그널

블로그의 포스팅 수와 날짜 분포는 "꾸준히 쓰는 사람"이라는 시그널이다.
시리즈를 하나로 합치면 밀도는 높아지지만 빈도가 손실된다.
시리즈는 유지하되 각 편의 독립 가치를 확보한다.

## B6. 메타데이터 (Jekyll Front Matter)

```yaml
---
title: "{제목}"
date: YYYY-MM-DD HH:MM:SS +0900
categories: [{메인 카테고리}, {서브 카테고리}]
tags: [{태그1}, {태그2}, ...]
---
```

카테고리·태그 목록은 발행 오케스트레이터(`devex:content-publish`)가 블로그 레포의 기존 값에서 재사용 우선 정책으로 선택한다.

## 필수 구조

```markdown
## TL;DR (B2)

```
