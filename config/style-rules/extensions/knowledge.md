# 지식 익스텐션

> **Base**: `base/readability.md` + `base/tone.md` + `base/punctuation.md` + `base/ai-tells.md`
> **Adds**: 개발 사전·노하우 특화 규칙 (저자 실무 판단 우선)

개발 사전·노하우는 개념을 간결하게 정리하되, AI가 생성 가능한 수준의 정의 나열은 가치가 없다.
**저자 실무 판단**이 들어간 비교·해석이 핵심이다.

---

## 적용 대상

| 매체 | 적용 |
|------|------|
| 위키 개발 사전 (재참조용) | 본 익스텐션 |
| 블로그 "기술 노하우 / 개발 사전" 카테고리 | 본 익스텐션 + `blog.md B6` 메타 |

---

## K1. AI 검증 질문

```
이 글에서 AI 가 쓸 수 없는 문장이 3개 이상인가?
```

3개 미만이면 분석(Analyze) 보강이 필요하다.
정의·비교표만으로는 합격선 미달이다.

---

## K2. 경량 PAAR

지식형은 PAAR 적용을 경량화한다.

- **Problem**: 이 개념을 왜 정리하는가 (실무에서 만난 맥락)
- **Analyze**: 유사 개념과의 차이를 **저자의 실무 판단 기준**으로 구분 (단순 정의 비교 금지)
- **Action**: 실무 적용 사례 (저자가 직접 설계·선택한 경험)
- **Result**: 생략 가능

---

## K3. 정의 비교는 표보다 판단 기준

Bad (정의 나열):
```markdown
| 개념 | 정의 |
|------|------|
| A | ... 입니다 |
| B | ... 입니다 |
```

Good (저자 판단 기준):
```markdown
| 상황 | 선택 | 이유 |
|------|------|------|
| {실무 상황 1} | A | 저자의 판단: ... |
| {실무 상황 2} | B | 저자의 판단: ... |
```

---

## 필수 구조

자유 형식.
다만 K1 검증 질문을 통과하려면 다음 요소가 1개 이상 포함되어야 한다.

- 저자가 직접 적용한 사례
- 저자가 직접 비교·선택한 판단 기준
- 저자가 직접 만난 함정 또는 우회
