## WR1. 데일리로그 기반 사실

기술하는 모든 성과는 **데일리로그 또는 검증 가능한 산출물(PR·티켓·문서·메트릭)** 에 근거한다.
"~로 성과를 거두었습니다" 식 모호 서술 금지.

Bad:
```
인프라 개선으로 효율을 높였습니다.
```

Good:
```
인증 미들웨어 분리 (PR #234·#240, 2026-Q1). 응답 시간 p99 380ms → 240ms (37% 감소, k6 측정).
```

`ai-tells A-1` (`을 통해`) 을 base 보다 강하게 차단 (성과서 모호화의 1순위 원인).

---

## WR2. PAAR 정량

각 성과 항목은 PAAR 구조로 정량 수치를 포함한다.

- **P**: 어떤 문제·요구가 있었는가
- **A**: 어떤 대안을 검토하고 무엇을 선택했는가 (저자 판단 근거)
- **A**: 무엇을 구현·수행했는가
- **R**: 수치 기반 성과

수치 없는 성과는 후순위 또는 누락.

---

## WR3. 조직 기여 + 개인 성장 분리

성과를 2축으로 분리한다.

| 축 | 예시 |
|----|------|
| 조직 기여 | 시스템 개선·장애 대응·온보딩 지원·문서화·표준화 |
| 개인 성장 | 새 기술 습득·자격증·외부 발표·사이드 프로젝트 |

---

## WR4. 시제·주어

- 시제: 평가 기간 성과는 과거형 (`~했습니다`)
- 주어 생략 원칙 유지 (`base/tone.md T13` 의 1인칭 회상)
- "저는" 금지 (`devex:content-write` 의 hard constraint 와 동일)

---

## WR5. 성과 어휘 회피

`ai-tells D-2` (`시사하는 바가 크다`), `D-3` (`주목할 만하다`) 등을 base 보다 강하게 차단.
조직 평가서 특유의 자기 부풀림 어휘 회피.

| 회피 | 대체 |
|------|------|
| `획기적인 성과` | 수치 + 비교 |
| `많은 기여` | 항목 수·시간·범위 |
| `최선을 다했습니다` | 무엇을 어떻게 |

## 필수 구조

```markdown
## 평가 기간: YYYY-MM-DD ~ YYYY-MM-DD

## TL;DR

- 핵심 성과 3-5건 (수치 포함)

## 조직 기여 (WR3)

### 성과 1: {제목}

**Problem**: ...
**Analyze**: ...
**Action**: ...
**Result**: {수치}

### 성과 2: ...

## 개인 성장 (WR3)

### 성장 1: {제목}

...

## 회고

{1~2 문단. 다음 평가 기간 목표}
```

PAAR 헤딩 라벨은 노출하지 말 것 (`tone.md T12`). 위 예시의 굵은 라벨은 평가서 작성용 가이드. 실제 본문은 자연어 흐름으로.
