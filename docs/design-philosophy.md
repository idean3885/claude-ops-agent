# 설계 철학 — AI를 내 원칙대로 길들이기

claude-devex 가 왜 이런 구조를 갖는지에 대한 기록이다. 기능 목록은 [README](../README.md) 에 있고, 이 문서는 그 뒤의 의도를 다룬다.

## 출발점

AI 에게 코드를 맡기면서 내 역할이 "코드 작성"에서 "의사결정과 검증"으로 옮겨갔다.

문제는 AI 가 매번 같게 동작하지 않는다는 점이다. 같은 요청에도 톤이 흔들리고, 어제 합의한 규칙을 오늘 잊고, 묻지 않은 수식어를 붙인다. 매번 사람이 교정하면 그 교정 자체가 일이 된다.

그래서 규칙을 내 기억이 아니라 하네스에 심기로 했다. 매 세션 시작, 매 파일 편집, 매 응답 직전에 규칙이 자동으로 걸리게 만든다. 사람이 기억해서 지키는 게 아니라, 환경이 규칙을 먼저 깔아둔다.

이것이 "AI 를 길들인다"의 뜻이다. 통제가 아니라, 합의한 규칙을 코드로 고정해 나와 AI 가 매번 같은 출발선에 서게 하는 것.

## 주입 지점 세 곳

규칙은 세 시점에 들어간다. 시점마다 막을 수 있는 실수가 다르다.

### 1. SessionStart — 세션을 시작할 때 환경을 맞춘다

`scripts/session-start.mjs` 가 세션이 열릴 때 한 번 실행된다.

| 동작 | 막는 실수 |
|------|-----------|
| provider 감지 (git remote host → 이슈 트래커 정의) | 트래커별 API 방식 추측 |
| git identity 자동 설정 (provider 기준) | 다른 계정으로 커밋 |
| 스타일 SSOT 미러 (`config/style-rules/` → `~/.claude/devex/style-rules/`) | 작성 규칙이 레포마다 제각각 |
| 스킬 트리거 컨텍스트 주입 | 자연어 요청이 워크플로우로 안 이어짐 |
| 버전 동기화 + git 자동 복원 | 캐시 디렉토리 버전 불일치 |

세션마다 환경이 같은 상태로 정렬되므로, 작업 중에 "이 레포는 규칙이 뭐였지"를 다시 떠올릴 필요가 없다.

### 2. hook — 매 편집·매 응답에 규칙을 건다

| hook | 시점 | 동작 |
|------|------|------|
| `forbidden-words-prompt.sh` | UserPromptSubmit | 금지 표현 룰을 system-reminder 로 사전 주입 |
| `forbidden-words-stop.sh` | Stop | 직전 응답을 정규식 매칭해 위반을 다음 턴에 통지 |
| `content-verify-posttool.sh` | PostToolUse (Edit/Write) | 문서 편집 후 AI 티·가독성·톤·구두점 자가 점검 유도 (opt-in) |
| `pre-tool-use.mjs` | PreToolUse | 퍼블릭 표면으로 가는 대외비 키워드 하드 차단 |

표현 가드는 응답을 출력 직전에 막거나 자동으로 고쳐 쓰지 않는다. 사전에 룰을 깔아주고(prompt), 사후에 위반을 알려준다(stop). 출력 직전 패턴 자가 대조는 여전히 어시스턴트의 의무로 남는다. 이렇게 한 이유는 [표현 가드](#표현-가드는-차단이-아니라-자가-대조)에서 다룬다.

### 3. 스킬 — 워크플로우를 강제한다

| 스킬 | 강제하는 것 |
|------|-------------|
| `flow` | 자연어 수정 요청 → 이슈 → spec → 구현 → commit → PR 순서 |
| `org-flow` | 멀티레포에서 사내/퍼블릭 provider 분기 |
| `content-write` → `content-verify` → `content-publish` | 작성 → 검증 → 발행 순서 |
| `cross-verify` | 멈추고 4축(의사결정·설계·문서·구현)을 확인하는 행위 |

스킬은 직접 호출하지 않는다. 자연어로 요청하면 트리거 키워드가 매칭되어 해당 워크플로우로 이어진다. 순서를 사람이 기억하지 않아도 되도록, 진입점을 자연어에 둔다.

## 왜 이렇게 하는가

### AI 티 제거

AI 가 쓴 한국어는 특정 패턴을 반복한다. 영어 번역투("~을 통해"), 근거 없는 단언("보장합니다"), 과시성 수식어("고수준의"), 의미 없는 형용사("원활하게"). 이런 표현이 섞이면 글이 사람이 쓴 것처럼 읽히지 않는다.

`config/style-rules/base/ai-tells.md` 가 이 패턴을 A~J 10대 카테고리로 분류한다. 분류 골격은 [`epoko77-ai/im-not-ai`](https://github.com/epoko77-ai/im-not-ai) (MIT) 에서 차용했고, 한국어 기술 블로그 맥락의 처방·예시는 자체 작성했다.

분류만으로는 안 지켜진다. 그래서 즉시 교정 가치가 있는 S1 패턴을 표현 가드 hook 의 정규식 룰로 등록한다. 분류 체계(SSOT)와 hook 룰이 카테고리 ID 로 1:1 매핑되어, 룰이 왜 존재하는지 역추적된다.

### 표현 가드는 차단이 아니라 자가 대조

처음엔 응답 출력 직전에 금지 표현을 막는 방식을 떠올린다. 하지만 출력 차단은 두 가지가 걸린다. 정규식이 문맥을 못 읽어 정상 표현을 오탐하고, 막힌 자리를 자동으로 메우면 의미가 틀어진다.

그래서 막지 않고, 사전 주입 + 사후 통지로 바꿨다. hook 은 룰을 미리 깔고 위반을 알려줄 뿐, 최종 판단은 어시스턴트가 출력 직전에 직접 대조해서 내린다. 도구가 사람의 판단을 대체하는 게 아니라, 판단할 자리를 좁혀준다.

### 저속한 표현 자제

기술 문서에 비격식·속어가 섞이면 포트폴리오로 쓸 때 신뢰가 깎인다. "땜빵", "박혀 있다" 같은 표현을 금지 룰로 등록해, 임시 조치는 "표면 수정/근본 미해결"로, 위치는 "명시/고정"으로 바꾸도록 사전에 안내한다.

### 이슈 플로우 강제

1인 개발이라도 변경 이력은 남겨야 추적된다. 자연어로 "이거 고쳐줘"라고 해도 `flow` 가 이슈 생성부터 PR 까지 끌고 간다. 모든 변경이 이슈 번호로 묶이고, 커밋·PR·버전 범프가 한 흐름에 들어온다. 변경 = 버전업 원칙도 여기서 강제된다.

## 자세

AI 를 길들인다는 건 AI 를 더 똑똑하게 만드는 게 아니다. 내 원칙을 코드로 고정해서, AI 의 출력이 매번 그 원칙 위에서 시작하게 만드는 것이다.

규칙이 사람 머릿속에 있으면 잊히고, 문서에만 있으면 안 읽힌다. 하네스에 있으면 매번 자동으로 걸린다. 그래서 규칙을 README 가 아니라 SessionStart·hook·스킬에 심는다.

이 자세는 도구가 바뀌어도 남는다. AI 모델이 바뀌고 더 좋아져도, "내 원칙을 환경에 고정한다"는 접근은 그대로 적용된다.
