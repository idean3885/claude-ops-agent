# 설계 배경과 의사결정 기록

claude-devex 가 왜 이런 구조인지, 어떤 선택지 중 무엇을 왜 골랐는지에 대한 기록입니다. 기능 목록은 [README](../README.md) 에 있고, 이 문서는 그 뒤의 결정을 다룹니다.

## 출발점

AI 에게 코드를 맡기면서 제 역할이 "코드 작성"에서 "의사결정과 검증"으로 옮겨갔습니다.

문제는 AI 가 매번 같게 동작하지 않는다는 점입니다. 같은 요청에도 톤이 흔들리고, 어제 합의한 규칙을 오늘 잊고, 묻지 않은 수식어를 붙입니다. 매번 사람이 교정하면 그 교정 자체가 일이 됩니다. 이 반복을 어떻게 줄일지가 아래 결정들의 공통 배경입니다.

## 결정 1. 규칙을 어디에 둘 것인가

| 선택지 | 문제 |
|--------|------|
| 사람이 기억한다 | 잊힌다 |
| 문서에 적어 둔다 | 안 읽힌다 |
| **하네스에 심는다** | 초기 구성 비용이 있으나 매번 자동 적용된다 |

세 번째를 골랐습니다. 규칙을 세 시점(세션 시작·파일 편집·응답 직전)에 자동으로 걸리게 두면, 작업 중에 "이 레포는 규칙이 뭐였지"를 다시 떠올릴 필요가 없습니다. 사람이 지키는 규칙은 안 지켜지는 순간을 막지 못하지만, 환경이 먼저 깔아 두면 저와 AI 가 매번 같은 출발선에 섭니다.

### 주입 지점 세 곳

**SessionStart**: `scripts/session-start.mjs` 가 세션이 열릴 때 한 번 실행됩니다.

| 동작 | 막는 실수 |
|------|-----------|
| provider 감지 (git remote host → 이슈 트래커 정의) | 트래커별 API 방식 추측 |
| git identity 자동 설정 (provider 기준) | 다른 계정으로 커밋 |
| 스타일 SSOT 미러 (`config/style-rules/` → `~/.claude/devex/style-rules/`) | 작성 규칙이 레포마다 제각각 |
| 버전 동기화 + git 자동 복원 | 캐시 디렉토리 버전 불일치 |

**hook**: 매 편집·매 응답에 규칙을 겁니다.

| hook | 시점 | 동작 |
|------|------|------|
| `forbidden-words-prompt.sh` | UserPromptSubmit | 금지 표현 룰을 사전 주입 |
| `forbidden-words-stop.sh` | Stop | 직전 응답 위반을 다음 턴에 통지 |
| `content-verify-posttool.sh` | PostToolUse (Edit/Write) | 문서 편집 후 자가 점검 유도 (opt-in) |
| `pre-tool-use.mjs` | PreToolUse | 퍼블릭 표면으로 가는 대외비 키워드 하드 차단 |

**스킬**: 워크플로우 순서를 강제합니다. `flow`(이슈 → spec → 구현 → commit → PR), `content-write → verify → publish`, `cross-verify`(멈추고 4축 확인). 직접 호출하지 않고 자연어 트리거로 이어지므로, 순서를 사람이 기억하지 않아도 됩니다.

## 결정 2. 표현 가드: 출력 차단인가, 자가 대조인가

처음엔 응답 출력 직전에 금지 표현을 막는 방식을 떠올렸습니다. 두 가지가 걸렸습니다. 정규식이 문맥을 못 읽어 정상 표현을 오탐하고, 막힌 자리를 자동으로 메우면 의미가 틀어집니다.

그래서 막지 않고 **사전 주입 + 사후 통지**로 바꿨습니다. hook 은 룰을 미리 깔고(prompt) 위반을 알려 줄 뿐(stop), 최종 판단은 어시스턴트가 출력 직전에 직접 대조해서 내립니다. 도구가 사람의 판단을 대체하는 게 아니라, 판단할 자리를 좁혀 줍니다.

## 결정 3. AI 티 분류: 자체 작성인가, 차용인가

AI 가 쓴 한국어는 특정 패턴을 반복합니다. 영어 번역투("~을 통해"), 근거 없는 단언("보장합니다"), 과시성 수식어("고수준의"), 의미 없는 형용사("원활하게") 같은 것입니다.

분류 골격은 [`epoko77-ai/im-not-ai`](https://github.com/epoko77-ai/im-not-ai)(MIT) 의 A~J 10대 카테고리를 차용하고, 한국어 기술 문서 맥락의 처방·예시는 자체 작성했습니다. 이미 검증된 분류 골격을 다시 짜는 대신 차용하고, 한국어 특화 부분에만 힘을 쏟는 선택입니다.

분류만으로는 안 지켜집니다. 그래서 즉시 교정 가치가 있는 S1 패턴만 표현 가드 hook 의 정규식 룰로 등록하고, 분류 체계(SSOT)와 hook 룰을 카테고리 ID 로 1:1 매핑해 룰이 왜 존재하는지 역추적되게 했습니다.

## 결정 4. 1인 개발에도 이슈 플로우를 강제할 것인가

1인 개발이면 이슈·PR 을 생략하고 바로 커밋하고 싶어집니다. 그래도 강제하기로 했습니다. 변경 이력이 이슈 번호로 묶여야 나중에 추적되고, "변경 = 버전업" 원칙이 흐트러지지 않습니다. 자연어로 "이거 고쳐줘"라고 해도 `flow` 가 이슈 생성부터 PR·버전 범프까지 한 흐름으로 끌고 갑니다.

## 이후 수정

첫 결정이 늘 옳지는 않았습니다. 실제로 되돌린 것들입니다.

- **릴리즈 자동화**: 초기 스크립트는 머지 후 main 에 직접 push 했습니다. 안전하지 않아, main 직접 push 를 제거하고 버전 범프를 PR 안에서 처리하는 방식(`post-merge-sync.sh`)으로 축소했습니다.
- **README**: 기능을 빠짐없이 나열하려다 장황해졌습니다. 가치 우선 구조로 두 번 재정돈(#109, #116)하고, hook·설정 상세는 `docs/` 로 분리했습니다.
- **톤 규칙**: 방어적·변명조 프레이밍과 과장을 막는 톤 규칙(T14)을 뒤늦게 추가했습니다.

규칙을 환경에 고정한다는 접근은 유지하되, 개별 규칙은 쓰면서 어긋난 지점을 찾아 고쳤습니다.

## 남는 것

도구는 바뀝니다. AI 모델이 더 좋아져도 "합의한 원칙을 환경에 고정한다"는 접근은 그대로 적용됩니다. 규칙이 사람 머릿속에 있으면 잊히고 문서에만 있으면 안 읽히지만, 하네스에 있으면 매번 자동으로 걸립니다. 그래서 규칙을 README 가 아니라 SessionStart·hook·스킬에 둡니다.
