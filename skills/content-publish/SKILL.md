# 블로그 발행

블로그 포스팅을 Jekyll(Chirpy 테마) 형식으로 변환하여 발행합니다.

## 금지 패턴 (HARD CONSTRAINTS)

| 금지 행위 | 이유 | 올바른 방법 |
|----------|------|------------|
| 변환 발행 모드에서 톤·내용 수정 | 입력 콘텐츠는 devex:content-write에서 검증 완료 상태여야 함 | devex:content-write 또는 오케스트레이터에서 선행 처리 |
| Phase 4 없이 Phase 5 진입 | 비평가 검토 누락 시 품질 미보장 | Phase 4 통과 후 Phase 5 진행 |
| em dash(`—`) 사용 | 한국어 키보드에 없는 문자, AI 생성 텍스트 신호 | 콜론(`:`), 쉼표(`,`), 괄호(`()`) 등 문맥에 맞는 대체 |
| 리더빌리티·톤 자체 검증 | devex:content-verify가 단일 소스, 자체 수행 시 기준 분산 | devex:content-verify(standalone) 위임 |
| 발행된 포스트의 파일명(slug) 변경 | URL이 깨져 이력서/외부 링크가 404 발생 | 제목(`title:`)만 변경, 파일명은 최초 확정 후 불변 |

## 역할

블로그 레포의 CLAUDE.md 포스팅 규칙에 맞는 포스팅을 작성하고, Jekyll 형식으로 변환하여 발행합니다.

## 트리거

- "블로그 작성", "글 쓰기", "블로그 변환", "블로그에 올려"

## 워크플로우 선택

| 모드 | 조건 | 설명 |
|------|------|------|
| **직접 작성** | 블로그 레포에서 실행, 콘텐츠 미제공 | Phase 1~7 전체 수행 |
| **변환 발행** | 콘텐츠가 이미 제공됨 (오케스트레이션) | Phase 4~7만 수행 |

---

## 설정

이 스킬은 **어떤 디렉토리에서든** 실행 가능합니다.
블로그 레포 경로는 글로벌 설정에서 로드합니다.

```
설정 파일: ~/.claude/blog-publisher.json
```

```json
{
  "blogRepo": "/path/to/blog-repo"
}
```

설정 파일이 없으면 사용자에게 경로를 질문하고 자동 생성합니다.

## 직접 작성 워크플로우

### Phase 1: 저자 소재 수집 및 주제 협의

**Step 1: 기존 포스팅 스캔**

`_posts/` 디렉토리의 기존 포스팅 제목과 주제를 확인한다.
중복/유사 주제가 있으면 사용자에게 알리고 차별화 방향을 협의한다.

**Step 2: 저자 소재 수집**

본문 작성 전에 저자의 고유 소재를 먼저 확보한다.
아래 항목 중 해당하는 것을 사용자에게 질문한다:

- 직접 겪은 문제와 해결 과정
- 실패/시행착오 사례
- 구체적 수치/측정 결과
- 선택의 이유 (왜 A를 선택하고 B를 버렸는가)
- 이전과 달라진 점 (Before/After)

> "아티클 정리" 카테고리의 경우, 소재 수집 항목을 다음으로 대체한다:
> - 이 아티클을 선택한 이유
> - 저자가 동의/반박하는 지점
> - 실무 적용 맥락 또는 계기
>
> 수집된 소재가 부족하면 본문 작성으로 넘어가지 않는다.
> "이 글은 저자만 쓸 수 있는가?"를 판단 기준으로 삼는다.

**Step 3: 주제 및 구조 확정**

수집된 소재를 바탕으로 사용자와 다음을 확정한다:

- 제목
- 카테고리: CLAUDE.md에 정의된 카테고리 구조를 따름
- 태그: 기존 태그 우선 재사용, 신규 생성 가능
- 본문 구조 (섹션 목차): 소재가 배치될 위치를 명시

### Phase 2: Front Matter 생성

CLAUDE.md에 정의된 Front Matter 규칙을 따른다.

- 필수 항목이 모두 포함되었는지 검증
- **날짜(`date`) 설정 규칙 (하드 규칙, 버그 방지)**:
  - 기준: **KST(+0900) 현재 시각**. 다른 타임존 사용 금지
  - 값: **현재 시각보다 과거**여야 한다. 임의의 미래 값(반올림 포함) 금지
  - 안전 버퍼: 배포 파이프라인 지연·워크플로우 큐잉을 고려해 **현재 시각 대비 최소 5분 이전** 권장
  - 검증: 커밋 직전 `date '+%Y-%m-%d %H:%M:%S %z'`로 현재 시각을 재확인하고 front matter date가 더 과거인지 확인
  - 배경: Jekyll 기본 `future: false` 설정에서 미래 date 포스트는 빌드에서 제외되어 배포 404 발생

### Phase 3: 본문 작성

CLAUDE.md의 작성 규칙과 마크다운 줄바꿈 규칙을 따른다.

- 코드 블록에 언어 명시
- 다이어그램 활용 가능

## 변환 발행 워크플로우 (Phase 4부터)

다른 스킬에서 콘텐츠를 넘겨받은 경우, Phase 1~3을 건너뛰고 Phase 4부터 수행합니다.

### 변환 발행 입력 검증 (GATE)

변환 발행 모드 진입 시, 입력 콘텐츠가 아래 조건을 충족하는지 확인합니다.
**충족하지 않으면 Phase 4로 진행하지 않습니다.**

| 항목 | 필수 | 설명 |
|------|------|------|
| 본문 | O | 정제된 마크다운 (대외비 제거 완료 상태) |
| 제목 | O | 포스팅 제목 |
| 카테고리 | O | `[메인, 서브]` |
| 태그 | O | `[태그1, 태그2, ...]` |
| 발행 날짜 | - | 미지정 시 **KST 현재 시각보다 최소 5분 이전**으로 설정 (Phase 2의 `date` 규칙과 동일). 미래 값 금지 |

## 공통 워크플로우 (Phase 4~7)

### Phase 4: 비평가 검토 (드래프트 점검 게이트)

본문 작성 후 반드시 아래 기준으로 검토한다.
이 검토는 발행 전 드래프트 점검 게이트다. 통과 전에는 드래프트 상태로 둔다.
통과하지 못하면 수정 후 재검토하거나 발행을 보류한다.

**품질 기준**

| 기준 | 검토 질문 |
|------|-----------|
| 주제 선명도 | 한 문장으로 요약되는가 |
| 단편 자립성 | 단편으로서 PAAR Analyze 깊이가 충분한가 (시리즈 분할로 얕아지지 않았는가) |
| 톤 적합성 | CLAUDE.md에 정의된 톤에 부합하는가 |
| 저자 관점 | 저자의 판단과 성장이 드러나는가 |
| 소재 활용 | 수집한 저자 소재가 본문에 반영되었는가 |
| 대체 불가성 | AI만으로 작성 가능한 내용이 아닌가 (저자 고유 경험이 핵심을 구성하는가) |
| 중복 여부 | 기존 포스팅과 겹치는 내용이 없는가 |
| 메시지 반복 | 같은 주장이 3회 이상 다른 섹션에서 반복되는가? 논증(근거 추가)이 아닌 단순 반복이면 제거 |
| 분량 적절성 | 기존 포스팅 평균 대비 ±50% 이내인가? 초과 시 반복/과잉 서술이 원인인지 확인 |

**리더빌리티 + 톤 검증: devex:content-verify 위임 (GATE)**

> **리더빌리티 24규칙 + 톤 검증은 devex:content-verify(standalone)에 위임한다. devex:content-publish가 직접 검증하지 않는다.**

devex:content-verify가 수행하는 검증:
1. 리더빌리티 24규칙 (콘텐츠 유형별 프로필 적용)
2. 저자 톤 (합쇼체, 주어 생략, 감정어 최소화)
3. 검증 리포트 반환

#### verify 결과 검증 (CHECKPOINT)

devex:content-verify 완료 후 아래 항목을 확인합니다.
**하나라도 실패 시 Phase 5로 진행하지 않습니다.**

- [ ] 리더빌리티 필수(●) 규칙 위반 0건
- [ ] 톤 위반 0건
- [ ] verify 종합 판정 = "게시 가능"

### Phase 5: Jekyll 변환

블로그 레포의 `CLAUDE.md`에서 포스팅 규칙(front matter, 스타일링 등)을 읽습니다.

**변환 규칙:**

| 항목 | 입력 | 블로그 |
|------|------|--------|
| 제목 | 본문 내 `# 제목` 또는 메타 | front matter `title`로 이동, 본문에서 제거 |
| TL;DR | `## TL;DR` 섹션 | `> **TL;DR**<br>` + `> 내용` + `{: .prompt-tip }` 로 변환. 라벨과 본문 분리, 아이디어 전환 지점에 `<br>` (B1 규칙) |
| 메타 섹션 | 하단 메타 블록 (있으면) | 제거 (front matter로 이동) |

**front matter:**

```yaml
---
title: "{제목}"
date: {날짜} {시간} +0900
categories: [{메인 카테고리}, {서브 카테고리}]
tags: [{태그}]
description: "{TL;DR 첫 문장}"
---
```

#### 시리즈 인덱스 포스팅

시리즈 상위 페이지는 블로그에서 **인덱스 포스팅(0번째 글)** 역할을 합니다.

| 항목 | 인덱스 포스팅 | 개별 포스팅 |
|------|-------------|-----------|
| 목적 | 프로젝트 전체 파악 (30초) | 특정 문제->해결 깊이 |
| 독자 | 면접관 (랜딩 페이지) | 관심 있는 기술 영역 독자 |
| 내용 | 프로젝트 요약 + 각 편 핵심 1줄 + 링크 | 완결된 문제->해결 스토리 |
| 갱신 | 시리즈 현행화 시 합본 역할 | 개별 수정 |

### Phase 6: 미리보기 및 파일 생성

변환된 전체 포스팅을 사용자에게 보여주고 확인을 받습니다.

블로그 레포의 `_posts/` 에 파일을 생성합니다.

- 파일명: `{YYYY-MM-DD}-{slug}.md`
- slug: 제목에서 키워드 추출, kebab-case
- **slug 불변 규칙**: 기존 포스트의 파일명(slug)은 절대 변경하지 않는다. `permalink: /posts/:title/` 설정에서 파일명이 곧 URL이므로, slug 변경 시 외부 링크(이력서, 포트폴리오)가 깨진다. 제목 변경은 front matter `title:`에서만 수행한다.

### Phase 7: 참조 프로젝트 등록 및 결과 보고

포스팅이 외부 프로젝트를 참조하는 경우, CLAUDE.md에 참조 프로젝트 섹션이 있으면 등록한다.

> 참조 프로젝트 섹션이 CLAUDE.md에 없으면 이 단계는 건너뛴다.

**결과 반환:**

| 항목 | 값 |
|------|---|
| 파일 경로 | `_posts/YYYY-MM-DD-slug.md` |
| 블로그 URL | `https://{site}/posts/{slug}/` |
| 발행 날짜 | `YYYY-MM-DD` |

## 기존 포스팅 편집

1. 포스팅 디렉토리에서 대상 포스팅 확인
2. 수정 사항 반영
3. 비평가 검토 (Phase 4 적용)
4. 참조 프로젝트 변경 여부 확인

## 참조 프로젝트 변경 추적

참조 프로젝트에 변경이 발생한 경우:

1. CLAUDE.md의 참조 프로젝트 테이블에서 영향받는 포스팅 확인
2. 참조 요소 기준으로 포스팅 업데이트 필요 여부 판단
3. 필요 시 포스팅 내용 수정

## 규칙

- 포스팅은 반드시 CLAUDE.md에 정의된 포맷을 준수한다
- 참조 프로젝트가 있는 포스팅은 CLAUDE.md에 등록한다 (참조 프로젝트 섹션이 있는 경우)
- 카테고리와 태그는 기존 값을 우선 재사용한다
- **내용 검토는 Phase 4(비평가 검토)에서만 수행**: 변환 발행 모드에서는 콘텐츠 품질이 입력 전에 해결되어야 함
