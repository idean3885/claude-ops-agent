# 콘텐츠 작성

마크다운 콘텐츠를 체계적으로 작성하는 범용 엔진입니다.
문서 성격 파악, 시리즈 구조 설계, 작성 원칙 적용, 인라인 검증까지 포괄합니다.

> **플랫폼 발행은 이 스킬의 범위 밖입니다.** 오케스트레이터(`dooray-wiki-write`, `devex:content-publish` 등)가 이 스킬을 호출하여 콘텐츠를 받고, 자체 발행 로직을 수행합니다.

## 금지 패턴 (HARD CONSTRAINTS)

| 금지 행위 | 이유 | 올바른 방법 |
|----------|------|------------|
| "저는/나는" 사용 | 저자 톤 위반: 주어 생략 서술이 원칙 | "~하고 있습니다", "~를 선택했습니다" |
| "~다/~이다" 한다체 혼용 | 합쇼체 통일 원칙 위반 | "~합니다/~입니다" |
| 감정어·수식어 남용 | 사실·행동·결과 중심 서술 원칙 위반 | 감정어 제거, 행동과 결과로 대체 |
| em dash(`—`) 사용 | 한국어 키보드에 없는 문자, AI 생성 텍스트 신호 | 콜론(`:`), 쉼표(`,`), 괄호(`()`) 등 문맥에 맞는 대체 |
| 권위 용어 남발 ("패턴/표준/베스트 프랙티스") | 공신력 없는 개념에 붙이면 저자가 제정하는 것처럼 보임 (T11) | 공신력 참조 있을 때만 사용, 그 외엔 "방식/접근법/구조"로 대체 |
| 짧은 부사절 뒤 관습적 쉼표 | 호흡 장식은 구두점 원칙이 아님 (PN1) | 끊어 읽어야 의미가 살 때만 쉼표 사용 |
| 세미콜론(`;`) 사용 | 한국어에 전통 없음, 영문 잔재 (PN5) | 마침표 또는 접속사로 문장 연결 |
| 리더빌리티·톤 자체 검증 | devex:content-verify가 단일 소스, 자체 수행 시 기준 분산 | devex:content-verify(inline) 위임 |

## 핵심 설계 원칙

### 원본 상세 → 목적별 가공

원본 콘텐츠는 상세할수록 좋습니다. 스펙, 대안, 트레이드오프, 수치, 출처까지 기록합니다.
발행 시점에 목적(블로그, 프레젠테이션 등)에 맞게 가공합니다.

> 수용하는 트레이드오프: 발행 시 변환 비용이 발생하지만, AI 자동화로 흡수.

### 단편 우선

콘텐츠 기본 단위는 단편입니다. 시리즈는 한 편으로 담기 어려운 연속 서사에서만 택합니다.

분할·시리즈를 택하기 전 점검합니다: 단편 1편으로 압축 가능한가. 가능하면 단편으로 작성합니다.
얕은 글을 시리즈로 늘리면 약한 편이 추가될 뿐입니다. 깊은 고찰이 담긴 단편이 양질의 글입니다.

### 시리즈 구조 (예외 케이스)

시리즈는 **인덱스(상위 문서)** + **개별 편(하위 문서)**으로 구성됩니다.

```
인덱스                     →    전체 파악 (30초 랜딩)
├── 1편                    →    독립된 문제→해결 스토리
├── 2편                    →    독립된 문제→해결 스토리
└── N편                    →    독립된 문제→해결 스토리
```

| 역할 | 인덱스 | 개별 편 |
|------|--------|---------|
| 목적 | 전체 구조 파악 | 특정 주제 깊이 |
| 독자 | 처음 방문자 (랜딩) | 관심 영역 독자 |
| 내용 | 개요 + 각 편 핵심 + 링크 | 완결된 문제→해결 스토리 |
| 갱신 | 편 추가/수정 시 | 개별 수정 |

### 개별 편 독립성

시리즈의 각 편은 **독립적으로도 읽힐 수 있어야** 합니다.

| 원칙 | 설명 |
|------|------|
| 독립 도입부 | 시리즈 순서 상관없이 이해 가능한 배경 설명 |
| 독립 제목 | "N편" 번호보다 주제가 먼저 드러나는 제목 |
| 독립 가치 | 각 편이 하나의 문제→해결 스토리를 완결 |

> 독립 가치가 낮은 편은 보강하거나 다른 편에 흡수를 검토합니다.

### 꾸준한 출력 시그널

포스팅 수와 날짜 분포는 "꾸준히 쓰는 사람"이라는 시그널입니다.
시리즈를 하나로 합치면 밀도는 높아지지만 빈도가 손실됩니다.
따라서 시리즈는 유지하되 각 편의 독립적 가치를 보장합니다.

## 절차

### Phase 1: 문서 성격 파악

콘텐츠 유형을 판별하여 이후 구조와 검증 기준을 결정합니다.

| 유형 | 판별 조건 | 필수 구조 |
|------|----------|----------|
| PoC | 기술 검증 목적, 코드 비중 높음, 결론이 "된다/안 된다" | TL;DR → 배경 → 환경 → 검증 → 결론 |
| 정보성 | 개념 설명/절차 안내, 독자가 따라할 수 있는 구조 | 개요 → 본문(절차/개념) → 정리 |
| 시리즈 어필 | 프로젝트 경험 기록, 포트폴리오용, 저자 판단이 핵심 | **PAAR**: TL;DR → 문제(P) → 분석(A) → 해결(A) → 결과(R) → 회고 |
| 지식 | 개발 사전/노하우, 기억용 간결체 | 자유 형식 (PAAR 면제) |

자동 판별이 어려우면 사용자에게 확인합니다.

#### 단편/시리즈 결정

기본은 단편(독립 글)입니다. 새 콘텐츠가 기존 시리즈에 속하거나, 단편으로 압축 불가능한
연속 서사일 때만 시리즈로 결정합니다. 시리즈 목록은 오케스트레이터가 제공합니다
(예: wiki-tree.md, 블로그 카테고리 등). 새 시리즈 신설은 예외 케이스로 신중히 판단합니다.

### Phase 2: 내용 작성

사용자와 대화하며 내용을 작성합니다.

#### 공통 원칙

| 원칙 | 설명 |
|------|------|
| TL;DR 필수 | 문서 상단에 핵심 요약 (1500자 이상 시) |
| 실무 사례 필수 | 이론만이 아닌 적용 맥락 포함 |
| 독립적 도입부 | 시리즈 내 글이라도 단독으로 읽힐 수 있는 배경 설명 |
| 말투 통일 | **~합니다/~입니다** (합쇼체). ~다/~이다 (한다체)와 혼용 금지 |
| 저자 톤 | 주어 생략 중심의 행동·결과 서술. "저는/나는" 사용하지 않고 "~하고 있습니다", "~를 선택했습니다" 형태로 작성. 개인 관점 표현 시 "~을 중요하게 생각합니다", "~라는 판단이었습니다" 등 간접 화법 사용. 감정어·수식어 최소화. |
| 구두점 최소주의 | 끊어 읽어야 의미가 살 때만 쉼표 사용. 시각적 호흡 장식은 구두점 역할이 아님. 세부 규정은 `readability.md § 9. 한국어 구두점 (PN1-PN5)` |

#### 유형별 작성 가이드

**PoC**: 결론 선행 + 재현 가능성이 핵심. 코드와 실행 결과를 분리 배치 (코드 → 결과 → 해석).

**정보성**: 따라하기 용이성이 핵심. 각 섹션이 독립적으로 참조 가능해야 함.

**시리즈 어필 (PAAR 구조 필수)**:
저자 고유 경험 + 의사결정 근거가 핵심. PAAR 흐름을 따릅니다.
- **문제(Problem)**: 왜 이 문제가 중요했는가
- **분석(Analyze)**: 어떤 대안을 검토했고, 왜 이 방법을 선택했는가 (핵심 섹션)
- **해결(Action)**: 무엇을 구현했는가
- **결과(Result)**: 수치 기반 성과
시리즈 5편보다 PAAR이 제대로 된 1편이 더 유의미합니다. 분석 근거가 빈약한 편은 다른 편에 흡수를 검토합니다.

### Phase 3: 인라인 검증 (사전 verify)

작성 완료 후, 발행 전에 콘텐츠 품질을 확보합니다.
별도 verify 패스 없이 **작성 Phase에서 즉시 적용**합니다.

#### 3-1. 비평가 검토

| 기준 | 검토 질문 |
|------|-----------|
| 주제 선명도 | 한 문장으로 요약되는가 |
| 톤 적합성 | 합쇼체로 통일되었는가. "저는/나는" 없이 주어 생략 서술인가. 감정어·수식어가 최소화되었는가 |
| 저자 관점 | 저자의 판단과 성장이 드러나는가 |
| 소재 활용 | 저자 고유 경험이 핵심을 구성하는가 |
| 대체 불가성 | AI만으로 작성 가능한 내용이 아닌가 |
| 메시지 반복 | 같은 주장이 3회 이상 단순 반복되는가 |
| 분량 적절성 | 유형 대비 ±50% 이내인가 |

#### 3-2. devex:content-verify 위임: 리더빌리티 + 톤 검증 (GATE)

> **이 단계는 devex:content-verify(inline)에 위임한다. devex:content-write가 직접 검증하지 않는다.**

devex:content-verify가 수행하는 검증 (devex:content-write는 관여하지 않음):
1. 리더빌리티 적용 (콘텐츠 유형별 필수/권장 구분)
2. 저자 톤 검증 (합쇼체, 주어 생략, 감정어 최소화)
3. 한국어 구두점 검증 (쉼표·콜론·세미콜론 규범, PN1-PN5)
4. 위반 항목 즉시 교정

devex:content-verify(inline)는 교정된 본문을 반환합니다.

#### verify 결과 검증 (CHECKPOINT)

devex:content-verify(inline) 완료 후 아래 항목을 확인합니다.
**하나라도 실패 시 본문을 반환하지 않습니다.**

- [ ] 리더빌리티 필수(●) 규칙 위반 0건
- [ ] "저는/나는" 본문 내 0회
- [ ] 한다체 문장 0건
- [ ] 세미콜론(`;`) 본문 내 0회 (코드 블록 제외)

### Phase 4: 메타데이터 프레임워크

콘텐츠에 발행 메타데이터를 부착합니다. 구체적 값(카테고리 목록, 태그 목록)은 오케스트레이터가 제공합니다.

```markdown
---

> **메타**
> 유형: {PoC | 정보성 | 시리즈 어필 | 지식}
> 구성: {단편 | 시리즈 N편}
> 카테고리: {메인}, {서브}
> 태그: {태그1}, {태그2}, ...
> 발행: {드래프트 | 발행 | 보류}
```
