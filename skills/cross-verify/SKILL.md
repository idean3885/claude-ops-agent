# 교차 검증 스킬

> 자동화가 아닌, **개발자가 의도적으로 멈추고 확인하는 행위를 구조화한 것**.
> 도구가 측정할 수 없는 "의미적 판단"에 집중한다.

---

## 호출 방식

- `/cross-verify` 직접 호출
- 트리거 키워드: "교차 검증", "크로스 체크", "cross verify"

## 프로젝트 설정 연동

devex 의 `config/cross-verify/profiles/` 디렉토리에서 프로젝트별 프로필을 자동 매칭한다.
프로필은 유저 스코프(`~/.claude/cross-verify/profiles/`)에 보관된다. 플러그인 업데이트와 독립적으로 유지된다.

### 프로필 탐색 절차

```bash
# 1. 유저 스코프에서 프로필 목록 탐색
ls ~/.claude/cross-verify/profiles/*.json 2>/dev/null
```

1. 프로필 파일을 순회하며 각 프로필의 `project` 필드를 읽는다
2. 현재 프로젝트의 CLAUDE.md 또는 디렉토리명에서 프로젝트 식별자를 추출한다
3. `project` 필드와 매칭되는 프로필을 자동 선택한다
4. 매칭 실패 시 사용자에게 프로필 목록을 제시하고 선택을 요청한다
5. 프로필이 하나도 없으면 범용 4축 검증을 실행한다 (기존 동작)

### 프로필 적용

프로필이 매칭되면:
1. `conventions` 경로의 문서를 읽어 컨벤션 기준으로 검증
2. `designDocs` 위치를 참조하여 설계 문서 확인
3. `focusAxes`에 지정된 축만 실행 (생략 시 전체)
4. `customChecks`의 항목을 해당 축 검증에 추가

> 프로필 관리: `~/.claude/cross-verify/profiles/`에서 직접 편집

## 동작 흐름

### Phase 1: 대상 파악

사용자가 제시한 컨텍스트를 수집한다.

**입력 유형:**
- 파일 경로 (코드, 문서, 설정)
- 커밋 범위 (`git diff`, `git log`)
- 설계 문서 또는 위키 URL
- 자유 텍스트 설명

컨텍스트가 불충분하면 사용자에게 질문한다:
```
검증 대상이 명확하지 않습니다. 다음 중 해당하는 것을 알려주세요:
- 검증할 파일 경로
- 관련 커밋 또는 PR
- 검토 중인 설계/의사결정 내용
```

### Phase 2: 4축 검증 실행

각 축별로 대상을 읽고, 구조화된 질문과 관찰을 생성한다.

---

#### 축 1: 의사결정 검증

> "왜 이 선택인가? 대안과 트레이드오프는?"

**검증 포인트:**
- 현재 선택의 근거가 명시되어 있는가?
- 검토한 대안이 있는가? 왜 기각했는가?
- **트레이드오프가 명시적인가?** — 이 선택으로 얻는 것과 포기하는 것이 기술되어 있는가?
- **반대 선택도 합리적인가?** (Opposite Test — 반대편이 사소하다면 이것은 진짜 선택이 아니다)
- 이 결정이 뒤집어질 수 있는 조건은 무엇인가?
- 결정의 영향 범위를 인지하고 있는가?
- **이 선택이 외부 공격에 노출되는 범위를 넓히는가?** 그 비용을 인지하고 수용하는가?
- **인증/인가 방식의 선택 근거는?** (예: API 키 vs OAuth, RBAC vs ABAC — 왜 이 방식인가?)

**관찰 방법:**
- 코드 주석, PR 설명, 커밋 메시지에서 의사결정 흔적 탐색
- 설계 문서가 있으면 "대안 검토" 섹션 확인
- 의사결정이 암묵적이면 그 자체를 지적
- 트레이드오프가 비용 없이 장점만 나열되어 있으면 "이것은 진짜 선택인가?"를 질문

**산출물:** 의사결정 근거 정리 — 명시된 것과 누락된 것, 그리고 수용한 비용을 구분하여 제시

---

#### 축 2: 설계 검증

> "엣지 케이스를 고려했는가?"

**검증 포인트:**
- 정상 경로(happy path) 외에 고려한 시나리오는?
- 실패 모드는 정의되어 있는가?
- 기존 정책/컨벤션과 일치하는가?
- 확장성/변경 용이성에 대한 고려가 있는가?
- 의존성 변경이 다른 모듈에 미치는 영향은?
- **외부에서 들어오는 입력을 신뢰하는 근거는?** 시스템 경계를 넘어올 때 검증 단계가 있는가?
- **실패 시 보안 상태는?** 인증/인가 서비스가 다운되면 요청을 거부하는가(안전), 허용하는가(위험)?
- **이 데이터의 민감도는?** 보호 수준이 그에 맞는가? (예: PII가 평문 로그에 노출되지 않는가)

**관찰 방법:**
- 코드의 분기 로직, 에러 핸들링 패턴 분석
- 기존 코드베이스의 유사 패턴과 비교
- 입력 경계값, null/undefined 처리 확인

**산출물:** 검증 포인트 목록 — 확인 필요한 엣지 케이스와 미고려 시나리오

---

#### 축 3: 문서 품질 검증

> "읽는 사람이 오해 없이 이해할 수 있는가?"

**검증 포인트:**
- 대상 독자가 명확한가?
- 용어가 일관되게 사용되는가?
- 정보의 구조가 논리적인가? (서론→본론→결론)
- 누락된 정보가 있는가? 과잉 정보는?
- 톤과 형식이 맥락에 적절한가?

**관찰 방법:**
- 문서/주석/커밋 메시지를 "처음 읽는 사람" 관점에서 검토
- 약어, 전제 조건, 암묵적 지식이 설명 없이 사용되는지 확인
- 코드 리뷰라면 PR 설명의 충분성 검토

**산출물:** 문서 품질 리포트 — 가독성, 완전성, 적절성 평가

---

#### 축 4: 구현 검증

> "코드가 설계를 정확히 반영하는가?"

**검증 포인트:**
- 설계/요구사항과 구현 사이의 갭은 없는가?
- 의도하지 않은 부수효과(side effect)가 있는가?
- 네이밍이 동작을 정확히 반영하는가?
- 테스트가 핵심 동작을 커버하는가?
- 하드코딩된 값, 매직 넘버가 있는가?
- **다른 사용자의 데이터에 접근할 때 소유권/권한을 확인하는 코드가 있는가?** (예: `/users/123` 호출 시 123이 본인인지 검증)
- **보안 관련 이벤트(로그인 실패, 권한 거부, 비정상 접근)가 기록되는가?**

**관찰 방법:**
- 설계 문서 ↔ 실제 코드 1:1 대조
- 함수명/변수명이 실제 동작과 일치하는지 확인
- 테스트 코드가 명세를 반영하는지 확인

**산출물:** 정합성 체크 결과 — 설계-구현 간 불일치 목록

## 핵심 원칙

### 하지 않는 것
- 코드, 문서, 설정을 직접 수정하지 않는다
- "이렇게 해야 합니다"라고 판단을 강제하지 않는다
- lint, test, type-check 등 기존 도구가 잡는 영역을 중복하지 않는다
- SAST/DAST가 탐지하는 기계적 보안 취약점(SQL 인젝션, XSS 등)을 점검하지 않는다
- 위협 모델링이나 보안 감사를 대체하지 않는다 — 보안 설계를 했는지 **상기시키는 역할**이다

### 하는 것
- 개발자가 놓쳤을 수 있는 관점을 제시한다
- 암묵적 가정을 명시적 질문으로 전환한다
- 의사결정의 근거와 맥락을 구조화한다
- 검증 과정 자체를 개발자가 함께 볼 수 있도록 투명하게 진행한다

### 기존 도구와의 관계
| 도구 | 역할 | 교차 검증과의 관계 |
|------|------|-------------------|
| lint/formatter | 문법·스타일 자동 교정 | 보완 — 스타일 너머의 의미를 다룸 |
| test/CI | 동작 검증 | 보완 — 테스트가 커버하지 않는 설계 의도를 다룸 |
| Ralph/ultrawork | 자동 구현·반복 실행 | 보완 — 실행 전후의 판단을 다룸 |
| code-review | 코드 품질 리뷰 | 보완 — 코드 너머의 의사결정·문서를 포함 |
| SAST/DAST | 구현 수준 보안 취약점 탐지 | 보완 — 도구가 잡을 수 없는 보안 설계 판단을 다룸 |
